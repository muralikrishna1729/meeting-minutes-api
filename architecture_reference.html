File-by-file responsibility map

FileLayerWhat it actually does
main.pypythonBuilds the FastAPI instance, registers all four routers under /api/v1, wires CorrelationIdMiddleware + CORSMiddleware + the slowapi rate limiter, and defines the lifespan context manager that logs startup info.
config.pypythonSettings(BaseSettings) — pydantic-settings class. Reads .env and validates every required variable at import time (SECRET_KEY, DATABASE_URL, REDIS_URL, CELERY_BROKER_URL, CELERY_RESULT_BACKEND, SUMMARIZER_TYPE, ACCESS_TOKEN_EXPIRE_MINUTES, REFRESH_TOKEN_EXPIRE_DAYS, HF_API_TOKEN). Missing any required field crashes the app on boot — this is what broke CI when env vars weren't injected.
models.pysqlThree SQLAlchemy ORM classes (schema below) plus a custom GUID TypeDecorator that stores UUIDs as native UUID on Postgres and CHAR(36) on SQLite, so the same model code works against both.
core/security.pypythonhash_password / verify_password (bcrypt via passlib). create_access_token / create_refresh_token — sign a JWT with sub (user id), type (access/refresh), jti (unique token id, used for blacklisting), exp, iat. decode_token verifies signature + expiry.
core/dependencies.pypythonget_current_user — FastAPI dependency that decodes the bearer token, checks Redis for a blacklisted jti, loads the User row. get_current_admin — same, plus asserts role == "admin", else 403.
core/cache.pyredisredis_client = aioredis.from_url(settings.REDIS_URL). Thin async wrappers: get_cache, set_cache(key, value, expire=3600), delete_cache.
core/logging.pypythonCustom JSONFormatter — emits {time, level, message, module, correlation_id} as a single JSON line per log entry. time is datetime.now(UTC).isoformat() (a raw datetime object here silently broke every log line for weeks — see the debugging playbook).
middleware/correlation.pypythonCorrelationIdMiddleware — generates a UUID per incoming request (or reuses an inbound X-Correlation-ID header), attaches it to the log record via context, and echoes it back on the response header.
routes/auth.pypython4 endpoints — see the endpoint reference table below for exact request/response shapes.
routes/minutes.pypython5 endpoints — upload fires the Celery task, the rest read/mutate MeetingMinutes rows scoped to current_user.id (IDOR prevention).
routes/admin.pypython2 endpoints, both gated by get_current_admin. /model/switch mutates the process-level summarizer selection at runtime — no restart needed.
routes/health.pypythonRuns SELECT 1 against Postgres and PING against Redis. Returns 503 if either fails, 200 otherwise.
services/summarizer.pypythonStrategy pattern — BaseSummarizer (abstract), MockSummarizer (rule-based, no network call), HuggingFaceSummarizer (POSTs to facebook/bart-large-cnn inference endpoint). get_summarizer() factory reads settings.SUMMARIZER_TYPE and returns the right instance.
services/celery_app.pyceleryCelery("meeting_minutes", broker=..., backend=..., include=["app.tasks.summarize"]). The include list is what makes the worker actually register the task on boot — without it the worker starts "successfully" with an empty task registry.
tasks/summarize.pycelerysummarize_meeting_task(self, meeting_id, original_text) — plain synchronous function (not async def, Celery workers don't await). Builds its own sync SQLAlchemy engine by taking DATABASE_URL and replacing asyncpg → psycopg2. Sets status processing → calls the summarizer → sets status completed/failed, with self.retry(exc=exc) on failure (max 3 retries, 60s delay).
db/session.pysqlcreate_async_engine(DATABASE_URL, pool_size=10, max_overflow=20). get_db() — FastAPI dependency, yields an AsyncSession per request, closed automatically after.
db/search.pysqlBuilds the raw @@ to_tsquery('english', ...) full-text query against the generated search_vector column, used by GET /minutes?q=.
alembic/env.pysqlOverrides configuration["sqlalchemy.url"] with settings.DATABASE_URL (sync-converted) before building the migration engine — this file originally ignored the app's real settings entirely and read a hardcoded value from alembic.ini instead.

Endpoint reference — exact request/response shape

POST /api/v1/auth/register no auth

Request body{"email": str, "password": str (min 8 chars)}
DB writeINSERT into users — checks email uniqueness first, 400 if taken
Response 201{"access_token": str, "refresh_token": str, "token_type": "bearer"}

POST /api/v1/auth/login no auth 5/min rate limit

Request body{"email": str, "password": str}
Logicbcrypt.verify() against stored hash → 401 if wrong or user inactive
Response 200Same shape as register

POST /api/v1/auth/refresh no auth, refresh token in body

Request body{"refresh_token": str}
LogicDecodes token, checks type == "refresh", re-fetches user, issues a new access token
Response 200{"access_token": new, "refresh_token": ..., "token_type": "bearer"}

POST /api/v1/auth/logout Bearer 5/min rate limit

LogicDecodes token → computes remaining_ttl = exp - now() → Redis SET blacklist:{jti} "1" EX remaining_ttl
EffectEvery future request with this exact token fails get_current_user's blacklist check, even though the JWT signature is still technically valid
Response 200{"message": "Logged out successfully"}

POST /api/v1/minutes/upload-text Bearer

Request body{"text": str} — raw transcript
DB writeINSERT meeting_minutes row, status='pending', user_id=current_user.id
Side effectsummarize_meeting_task.delay(meeting_id, text) — fires onto Redis broker db1, does not block
Response 202{"id": uuid, "status": "pending"} — returned before summarization runs

GET /api/v1/minutes/{id} · /{id}/status · ?q=keyword Bearer

Ownership checkEvery query includes WHERE user_id = current_user.id AND deleted_at IS NULL — a different user's ID returns 404, not 403 (avoids confirming the record exists at all)
?q= searchRuns the GIN-indexed search_vector @@ to_tsquery(...) query from db/search.py

DELETE /api/v1/minutes/{id} Bearer

LogicUPDATE meeting_minutes SET deleted_at = now() — never a physical DELETE

GET /admin/stats · POST /admin/model/switch Bearer, role=admin

statsAggregate counts of meeting_minutes grouped by status
model/switchBody {"summarizer_type": "mock"|"huggingface"} — mutates the running process's summarizer selection, no restart required

Full request-by-request workflows

Workflow A — Upload → async summarization → poll

1
Client → Nginx → FastAPI

POST /api/v1/minutes/upload-text with a bearer access token and {"text": "..."}".

2
Auth dependency runs first

get_current_user decodes the JWT, checks Redis GET blacklist:{jti} — if present, 401 before the route body ever executes.

3
routes/minutes.py inserts the row

INSERT INTO meeting_minutes (id, user_id, original_text, status) VALUES (uuid4(), current_user.id, text, 'pending'), then await db.commit().

4
Task enqueued, response returned immediately

summarize_meeting_task.delay(meeting_id, text) serializes the call onto Redis db1 (the Celery broker). FastAPI returns 202 without waiting — total time is milliseconds.

5
Celery worker (separate OS process) dequeues it

The worker was already running, connected to Redis db1, with summarize_meeting_task registered via celery_app.py's include=[...] list.

6
tasks/summarize.py opens its own sync DB session

Not the same async engine FastAPI uses — a separate sync_engine built from DATABASE_URL with the driver swapped to psycopg2, because Celery's task function is plain def, not async def.

7
Status flips to processing, commit

Then get_summarizer().generate(original_text) runs — either the mock (instant, rule-based) or a real HTTP POST to HuggingFace's inference API.

8
Result written back — PostgreSQL is the only source of truth

summary, action_items, decisions are set, status='completed', committed. Celery's own Redis result backend (db2) is not what the API reads from — only the DB row is authoritative, by design.

9
Client polls

GET /api/v1/minutes/{id}/status repeatedly — same auth + ownership check as step 2-3, just a SELECT instead of an INSERT, until status == "completed".

Workflow B — Auth lifecycle: token issuance → use → rotation → revocation

1
Register or login

Password checked/hashed with bcrypt. On success, two JWTs are minted: an access token (ACCESS_TOKEN_EXPIRE_MINUTES, default 60 min) and a refresh token (REFRESH_TOKEN_EXPIRE_DAYS, default 7 days). Each carries its own jti (unique ID) and type claim.

2
Every protected request

Authorization: Bearer <access_token>. get_current_user verifies the signature with SECRET_KEY, checks exp hasn't passed, checks the jti isn't in Redis's blacklist set, then loads the User row by sub.

3
Access token expires

Client calls /auth/refresh with the refresh token instead of logging in again. The endpoint verifies type == "refresh" specifically — an access token can't be used here even if otherwise valid, preventing privilege confusion between the two token types.

4
Logout — the interesting part

JWTs are stateless by design — the server can't normally "delete" one. The workaround: compute how much time is left on the token (exp - now()) and write its jti into Redis with that exact TTL as a blacklist entry. The blacklist entry expires at the same moment the token would have expired anyway, so Redis never accumulates stale blacklist data.
